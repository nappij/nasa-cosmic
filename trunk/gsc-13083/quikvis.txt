*************************************
*      SECTION 1. INTRODUCTION      *
*************************************


QUIKVIS computes target availability information for fixed and moving
celestial targets.  The results are useful in selecting candidate
targets, examining the effects of observation requirements, and doing
gross assessments of the effects of the orbit's right ascension of the
ascending node(RAAN).  The results are not appropriate for scheduling
purposes(see LIMITATION note at the end of this section).

Target availability means the earth does not block the line of sight to
the target and all observation requirements are satisfied. 


The user specifies:

      - orbit altitude and inclination

      - range of orbit right ascension of the ascending node

      - range of dates of interest

      - targets to be observed(individual targets or a sky survey)

      - observation requirements;  during an observation, none, some, or
        all of the following may be required....

           s/c is in orbit night

           the separation angle between the target direction and the
               velocity vector is greater than a specified minimum

           the separation angle between the target direction and the
               earth horizon is greater than a specified minimum

           the separation angle between the target direction and the
               zenith direction is less than a specified maximum






                         ( CONTINUED )







QUIKVIS produces target availability information as follows...

      - for individually specified targets,

          a table showing the duration per orbit that the target is
          available;  the durations are given as a function of date and
          RAAN or RAAN's mean solar time;  (see section 3, example 1)

          detail output showing the mean anomalies where individual
          observation requirements are satisfied;  the mean anomalies
          are given as a function of date and RAAN or RAAN's mean solar
          time;  (see section 3, examples 2 and 3)

          detail output showing when, relative to one of several orbital
          events available, observation requirements are satisfied;
          (see section 3, example 4)

          a data file of the above-mentioned detail output;  (see
          section 10, table 6, parameter named LEVDETAIL;  also see
          section 12 for the format definition)

      - for a sky survey,

          for each date, a table showing the minimum duration targets
          are availabile per orbit;  the table is given as right
          ascension -vs- declination and the duration is for a target at
          that table location;  the mimumum is the smallest of the
          durations computed for the individual RAAN's;  (similar to
          section 3, example 5, but for one date)

          a series of tables, each associated with a time in the
          specified range;  each table is a composite of single-date
          tables described above;  the composite duration is the
          minimum of each target's single-date minimum, considering all
          dates from the first to the current one;  (see section 3,
          example 5);

          corresponding to the two items immediately above, maps of the
          sky showing the duration regions are available;  (see
          section 3, example 6);



LIMITATION
----------

Because of the models involved, this program's output should not be used
when high accuracy is needed.  In particular, it is inappropriate for
scheduling actual mission operations.  This program uses the following
models: 

    - the spacecraft's trajectory is Keplerian and circular
    - the earth is spherical
    - sun and planet positions are determined analytically
    - user specifies the orbital parameters in Mean of 1950.0
      coordinates
    - potential target occultation by the moon or planets is ignored
    - atmospheric refraction is ignored




************************************
*    SECTION 2. NOTABLE OPTIONS    *
************************************



1) Specifying the orbit RAAN.  The user may specify the RAAN range
   directly or in terms of its mean solar time.  (see table 4)

   The latter ties the RAAN to a launch time of day and may be useful
   for missions where the launch window is constant over a range of
   dates.  The mean solar time parameters are used to compute the
   RAAN values for the dates involved.
   


2) Sky survey mode.  If the payload has potential targets throughout the
   sky, this program can be used in a survey mode to identify promising
   regions of the sky.  One selects the survey option by not specifying
   any targets on the observation requirements file.  Examples 5 and 6
   of section 3 show the kind of output produced by this option.
   


3) Detail output.  For a run that processes individually specified
   targets, one can get the following(controls input via table 6 of
   section 10)....

      - a table showing, for each RAAN or RAAN mean solar time, the
        mean anomalies where individual observation requirements begin
        and cease to be satisfied; a table is given for each date;  the
        mean anomalies are relative to a user-specified orbit event set
        in table 8 of section 10;  see example 2 in section 3


      - for each RAAN or RAAN mean solar time, the above information is
        presented in bar graph format to illustrate the overlapping of
        requirement satisfaction periods;  each requirement has an
        associated bar;  one set of this output is given for each date;
        the mean anomalies are relative to a user-specified orbit event
        set in table 8 of section 10;  see example 3 in section 3

      - a table in bar graph format showing the time after a
        user-specified orbit event(table 8) when all requirements are
        satisfied;  there is one bar per RAAN;  a table is given for
        each date;  see example 4 in section 3

      - a data file containing the data values used to generate the
        table and bar graph output mentioned above;  see section 12
        for the file contents and format


4) XYPLOT data files.  The program can create two data files used by the
   TOSS XYPLOTTER program to display target availability as a function
   of the orbit RAAN or the RAAN's mean solar time.  (see section 11 and
   table 6 in section 10)

      $! QUIKVIS.COM

      $ DEASSIGN FOR001

      $ DEASSIGN FOR027

*******************************
*   SECTION 6. PROGRAM FLOW   *
*******************************


Before each of the read instructions which will be discussed in
section 10 of this user guide, the user is prompted for the input
values.  Except for the first prompt, the current values of the
variables involved are displayed on unit 6 so the user can decide on
changes.  This occurs even if the input is from a file and no real-time
decision is involved. 


ERROR CHECKS ON USER INPUT ARE DONE.


FLOW:

   STEPS 1 AND 2 ARE EXECUTED ONCE PER RUN. THE OTHERS FORM A LOOP.


   1. The program initializes its internal arrays and default values.
      It reads the observation requirements file and if necessary, the
      target catalogue.  An error in either file ends the run.


   2. The USER SEES a prompt message asking whether this is an
      interactive run.  The reply affects the response to user input
      errors and the generation of progress reports during execution.
      The program believes the user's response, regardless of the actual
      nature of the run.  (Table zero)


   3. The USER SEES a series of prompt messages asking for parameters
      controlling the program.  If an error occurs here, the program
      reprompts(interactive runs) or ends(non-interactive). (Tables 2-8)


   4. The USER SEES a display of parameters controlling the program.
      The current case executes.  If this is an interactive run,
      messages are sent occasionally to the prompt unit so the user can
      monitor the run's progress.


   5. The program repeats steps 3 and 4 until the user's input contains
      an end of file or until a fatal error is sensed.


********************************
*    SECTION 7. USAGE NOTES    *
********************************



LIMITATIONS AND COMMENTS
------------------------

   - all orbits are circular;  extending the program to handle eccentric
     orbits is not difficult because the underlying algorithms handle
     eccentric orbits and QUIKVIS has been coded with eccentricity and
     argument of perigee variables present but set to zero

   - all orbit computations assume Keplerian motion

   - Mean of 1950.0 coordinates are used, including those for the
     s/c orbital parameters

   - observation requirements are the same for all targets in the run

   - the sky survey option is very time consuming;  you should do an
     initial run using only one date to verify the correctness of your
     input and get a timing estimate;  test runs have varied according
     to the load on the computer(VAX 11/780), but 20 minutes elapsed
     time per date is not unusual with 37 orbit RAAN's involved;
     efficiency modifications will be made in the future if program
     usage warrants the effort;  subroutine TAVISPAK could use a mod
     specifically for circular orbits(internally, QUIKVIS uses a very
     small eccentricity to avoid ambiguity about the perigee location)
     
   - when used as targets, the sun, moon, and planets, have zero radius;
     when used for shadow computations, the sun has its correct radius

   - the earth is the only body that can occult a target;  for example,
     if a target is behind the moon, this program will not sense it

   - the earth is spherical

   - the program can handle up to 703 targets(703 accommodates the
     sky survey option)

   - the program can handle up to 361 orbit RAAN's

   - the program can handle up to 100 dates in the date range

   - the target catalogue may have at most 5 parameters in the
     set of parameters holding the location and miscellaneous data;
     described section 8

   - only 5 kinds of observation requirements permitted;  the associated
     keywords in the observation requirements file are DAYNIGHT, 
     BRIGHTERT, DARKERT, VELAVOID, ZENITH;  furthermore, the bright and
     dark horizon avoidance angles must be equal; described in section 9
     
   - the detail output option can create large output files and should
     be used judiciously;  see comments in section 12 for space
     requirements


ERROR HANDLING
--------------

   If errors are sensed in the target catalogue or observation
   requirement file, the run ends.

   In the interactive mode the program responds to user input (unit 5)
   errors by reprompting for the input.  If not interactive, a user
   input error ends the program.

   Throughout the code, error checks are done in places where errors are
   unlikely except through careless modification in the future.  Not all
   such errors can be anticipated, but those that were will cause the
   program to stop with a message.

*   SECTION 9. INPUT FILE - OBSERVATION REQUIREMENTS   *
********************************************************


The following is an example of an observation requirements file.

        QUIKVIS OBS REQMTS.
        'EXAMPLE'/
        'DAYNIGHT',1/
        'VELAVOID',90./
        'ZENITH',95./
        'ENDREQ'/
        23/
        107/
        -9999/



The TOSS document OBSREQMT.TXT fully describes the file's format. An
abbreviated description follows and reflects both QUIKVIS and catalogue
reader requirements.

The first line is a comment line. It must be present but its contents
are ignored.

The second line defines the name of the experiment.  It is used for
output purposes and has no other significance.

The next several lines, up to the ENDREQ keyword, define the
requirements that must be satisfied during an observation.  QUIKVIS
allows five other keywords and each has one accompanying value.
Keywords must be enclosed in single quotes.   If a keyword is absent,
the observation requirement takes the default value.

      KEYWORD   VALUE
      ------    -----

      DAYNIGHT    0   orbit night not required (default) 
                  1   orbit night required

      BRIGHTERT   x   x= the target's minimum angle above the
                      sunlit horizon; in degrees;  default = 0.0

                      must be zero or positive

      DARKERT     x   x= the target's minimum angle above the
                      shadowed horizon;   in degrees;   default = 0.0

                      must be zero or positive

                      WARNING: QUIKVIS requires that the horizon angles
                               be equal

      VELAVOID    v   v= minimum angle between the line of sight to the
                      target and the velocity vector;  in degrees;
                      default = 0.0

                      must be in 0 to 180 range

      ZENITH      z   z= maximum angle between the line of sight to the
                      target and the zenith direction;  in degrees
                      default = 180.0

                      must be in 0 to 180 range

      ENDREQ    none  must be present and the last keyword.






                     ( continued )




                   ( continuation )




Following the keywords, zero to 703 target ID's are supplied.  A record
containing -9999 in the ID location signals the end.  If no ID's are
supplied, QUIKVIS executes the sky survey option.  ID's need not be in
ascending order.  Zero and negative ID's are ignored.  Duplicate ID's
are treated as a single ID. 




DISCUSSION:

      zero, some, or all may be present

      DARKERT and BRIGHTERT must be both present or both absent

      since QUIKVIS uses circular orbits, ZENITH and DARKERT/BRIGHTERT
      have similar effects;  horizon avoidance is the more efficient
      because it is used in the earth occultation algorithm

***************************************************************
*    SECTION 12. DEFINITION OF THE DETAIL DATA FILE FORMAT    *
***************************************************************


   The file is written by subroutines QUIKVIS5A3, QUIKVIS5A2C, and
   QUIKVIS5A2C3.




The Detail Data File, unit 27, is generated when LEVDETAIL(section 10,
table 6) is non-zero.  There are four levels of detail.  The file
contains standard header records plus the following FOR EACH TARGET
AND EACH DATE:


     detail level 1 (minimum detail):

         a) tells when the target is guaranteed to be availabile;
            this is the availabilty period common to all RAAN's


     detail level 2:

         items a and...

         b)  tells when the individual observation requirements are
             guaranteed to be satisfied;  these periods are common to
             all RAAN's


     detail level 3:

         items a, b, and...
         
         c)  for each RAAN processed, tells when the target is
             available;


     detail level 4 (maximum detail):

         items a, b, c, and...

         d) for each RAAN processed, tells when individual observation
           constraints are satisfied


The file is written so that a post-processing program can read it in a
list-directed manner.  If QUIKVIS is executed with multiple cases and
more than one case creates a detail data file, a single file is
produced with the information from one case following that of a
previous case.


STORAGE REQUIREMENTS(within about a percent, in 512-byte blocks)
--------------------

   level 1:

      0.20  *         1         *  number of dates  *  number of targets

   level 2:

      0.40  *         1         *  number of dates  *  number of targets

   level 3:

      0.19  *  number of RAANS  *  number of dates  *  number of targets

   level 4:

      0.35  *  number of RAANS  *  number of dates  *  number of targets




HEADER RECORDS(records are the same for all levels of detail)
--------------


The header records give information about parameters used by QUIKVIS to
generate the data present in this file.


CAUTION:

   if you write a program that reads this file, be aware that
   the number of header records may change in the future if there is a
   need and the number of observation requirements may also change.
   Therefore, use the values you read from records 1 and 7 instead of
   fixed values(13 and 4 are current at this writing).  If new
   requirement types are added to QUIKVIS, new records will be added
   after record 11 and new angle pairs will be added in the
   node-specific detail data records(see items h and j).  If additional
   information about QUIKVIS parameters is needed, they will be added
   after record 13.


DESCRIPTION:

   rec 1 (1 integer value)

            a) number of header records following this record;
               currently, there are 13 header records following the
               first one, so the value written in this record is 13

   rec 2 (4 real values) - orbit parameters

            a) SMA(km)   b) ECC   c) INCL(deg)   d) ARGP(deg)

   rec 3 (1 real value)

            a) orbit period(Keplerian) in minutes

   rec 4 (1 integer value, 2 real values, 1 integer value) - node info

            a) flag indicating how orbit nodes are specified
               =1, RAAN specified directly
               =2, mean solar time of RAAN is specified
            b) first RAAN(deg) or RAAN mean solar time(hrs),
            c) increment(deg or hours)
            d) number of nodes

   rec 5 (2 real values, 1 integer value) - date info

            a) first date and time processed(YYMMDD.HHMMSS)
            b) increment(days)
            c) number of YYMMDD.HHMMSS's processed

   rec 6 (1 integer value)

            a) number of targets processed





                   ( continued )




                  ( continuation )


  rec 7  (1 integer value)

            a) number of observation requirement data records following
               this one;  currently, QUIKVIS handles 4 types of
               requirements, so the value in this record is 4

  rec 8  (1 logical value, 1 real value)

            a) T/F flag indicating whether earth horizon avoidance was
               considered;  T = considered;   THIS IS ALWAYS TRUE
            b) avoidance angle(deg)

  rec 9  (1 logical value)

            a) T/F flag indicating whether orbit night is an
               observation requirement;  T = yes,  F = day/night ignored

  rec 10 (1 logical value, 1 real value)

            a) T/F flag indicating whether velocity vector avoidance was
               considered;  T = considered
            b) avoidance angle(deg)

  rec 11 (1 logical value, 1 real value)

            a) T/F flag indicating whether maximum zenith vector
               separation was considered;  T = considered
            b) max separation angle(deg)

  rec 12 (1 integer value)

            a) indicator identifying the orbit event to which relative
               mean anomalies are referenced

               = 1 s/c crosses orbit sunset
               = 2 s/c crosses orbit sunrise
               = 3 s/c crosses sun meridian
               = 4 s/c crosses ascending node
               = 5 s/c crosses descending node
               = 6 s/c crosses target meridian
               = 7 s/c crosses zero right ascension

  rec 13 (1 integer value)

            a) indicator identifying the detail level for this file
               1 = minimum, 2 and 3 = intermediate, 4 = maximum

  rec 14 (1 integer value)

            a) the value zero;  the last header record;  present as a
               visual separator between the header and the detail data
               portion of this file so a person can read it more easily



DETAIL DATA RECORDS
-------------------


Detail data records are created as follows(a record has 2 or more lines
of text when listed): 


1. FOR EACH TARGET, DO....

      2. FOR EACH DATE, DO....

             3. IF LEVEL 3 OR 4 IS WANTED:

                  4. AS EACH RAAN IS PROCESSED, DO....

                        WRITE AVAILABILITY INFO FOR THE CURRENT RAAN
                          (if level is 4, include info about
                           individual obs requirements)

                        END OF INNER BLOCK 4

                  END OF INNER BLOCK 3

             WRITE A GUARANTEED AVAILABILITY DATA RECORD
               (if level is greater than 1, include info telling when
                individual requirements are guaranteed to be satisfied)

             
             IF LEVEL IS 3 OR 4, WRITE A VISUAL SEPARATOR RECORD

             END OF INNER BLOCK 2

      END OF OUTER BLOCK 1



DATA DESCRIPTION:


   a) integer value;  this is a counter identifying the target index
      within the run(ie, the first, second, third, ... target processed)

      for levels 3 and 4:

             visual separators are present in the file;  the value of
             item a is 0 in separator records;  in separator records,
             the remaining data items are absent


   b) integer value;  this is the target ID as specified in the
      requirements file and as read from the target catalogue;


   c) real value;  this is the date and time associated with this
      record;  format is YYMMDD.HHMMSS




                   ( continued )


                  ( continuation )


   d) real value;  this is the orbit RAAN associated with this record;
      in degrees;

      for levels 1 and 2:

             value is always 999.000;  the only records present are
             those giving guaranteed availability info

      for levels 3 and 4:

             this value is 999.000 for the guaranteed availability
             records

             if not 999.000, value is the RAAN value for for which
             RAAN-specific availability info is being given in this
             record


   e) real value;  this is the target availability duration for the
      date and time in item c and the RAAN value in item d;  in minutes;

      if this is a guaranteed availability record(ie, item d value is
      999.000) then the duration is for the guaranteed availability for
      this date and for all nodes processed;  the accuracy is 1/3600 of
      the orbital period


   f) real value;  this is the mean solar time of the RAAN in item d;
      in hours

      if value is 999.000, then so is item d and this is a guaranteed
      availability record


   g) real value;  this is the mean anomaly at which occurs the orbit
      event identified in header records ;  in degrees;

      if value is 999.000, then so is item d and this is a guaranteed
      availability record


   h) integer value;  this is the number of angle pair data groups
      present in this record(items i-m):

                level   item h value
                -----   ------------
                  1          1
                  2          5
                  3          1 in RAAN-specific records
                  3          5 for guaranteed availability records
                  4          5

      the 5's are based on the current number of observation
      requirements being 4; see the caution note before the header
      record description in this section



                   ( continued )



                  ( continuation )


   i) first angle pair data group(always present):

      integer value(=N1) followed by N1 pairs of real values;

      N1 = the number of periods during the orbit when observation
           requirements are satisfied simultaneously;  the periods are
           defined by a pair of angles;  each angle is a mean anomaly,
           relative to the orbit event identified in the header records

      the first angle value in a pair tells when satisfaction begins,
      the second tells where it ends;  the first value is always less
      than or equal to the second, and may be negative

      for RAAN-specific data records, the pairs give availability info
      only for the RAAN involved;  for guaranteed availability data
      records(ie, item d is 999.000), the pairs give availability when
      all RAAN's have been considered;

      for RAAN-specific data records, the values are accurate as
      written; for guaranteed availability records, the values are
      accurate to 0.1 degrees;


   j) second angle pair data group(present when item h greater than 1):
      (this is horizon avoidance info)
      
      integer value(=N2) followed by N2 pairs of real values;

      N2 = the number of periods during the orbit when horizon avoidance
           requirement is satisfied;  the periods are defined by a pair
           of angles;  each angle is a mean anomaly relative to the
           orbit event identified in the header records

      the first angle value in a pair tells when satisfaction begins,
      the second tells where it ends;  the first value is always less
      than or equal to the second, and may be negative

      for RAAN-specific data records, the pairs tell when the constraint
      is satisfied for the RAAN involved;  for guaranteed availability
      data records(ie, item d is 999.000), the pairs give the periods
      common to all RAAN's processed

      if there was no requirement, one pair is present with values
      0. and 360.


   k) third angle pair data group(present when item h greater than 1):
      (this is orbit night info)
      
      integer value(=N3) followed by N3 pairs of real values;

      same description as item j, but the constraint is orbit night


   l) fourth angle pair data group(present when item h greater than 1):
      (this is velocity vector avoidance info)
      
      integer value(=N4) followed by N4 pairs of real values;

      same description as item j, but the constraint is velocity vector
      avoidance


   m) fifth angle pair data group(present when item h greater than 1):
      (this is zenith direction separation info)
      
      integer value(=N5) followed by N5 pairs of real values;

      same description as item j, but the constraint is max zenith
      zeparation
